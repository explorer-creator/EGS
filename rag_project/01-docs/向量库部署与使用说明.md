# 向量数据库与前后端交互对接说明

本文档详细记录了本地轻量级向量数据库的部署流程，以及前端 Vue 组件与后端 Spring Boot 的完整交互链路。

## 第一部分：向量数据库 (Qdrant) 部署与使用

为了支持 RAG（检索增强生成）功能，本项目引入了 Qdrant 作为本地向量数据库。它采用无依赖的容器化部署，资源占用极小，非常适合在有限的硬件条件下或 Ubuntu 24 等常规桌面系统中进行高效开发与测试。

### 1. 一键启动与持久化部署

请在项目根目录（即 `docker-compose.yml` 所在位置）打开终端，确保 Docker 环境正常可用，然后执行以下命令：

```bash
# 后台一键启动 Qdrant 服务
docker compose up -d

# 查看实时运行日志（调试用）
docker compose logs -f qdrant

# 停止服务（不删除数据）
docker compose down

数据持久化机制：
服务启动后，底层索引和向量数据会自动挂载并存储在项目根目录的 EGSfb/vector-data/ 文件夹下。请勿手动修改该目录内的文件。若需要彻底清空并重置数据库，只需停止服务后删除该文件夹即可。

2. 本地接口暴露与可视化面板
服务成功运行后，会在本地暴露以下两个核心端口供调用：

REST API (HTTP) 端点 (http://localhost:6333)

用途：适合浏览器直接访问、简单的 HTTP 连通性测试。

可视化控制台：在浏览器中访问 http://localhost:6333/dashboard 即可打开自带的 Web UI，能够直观地查看现有的 Collection 以及存入的文本向量数据。

gRPC 端点 (localhost:6334)

用途：强烈推荐后端的 Java Spring Boot 核心业务使用此端口进行连接。gRPC 协议的通信效率和并发处理能力远优于普通 HTTP。

3. Java 后端对接准备
当前的 LlmController.java 代码中已经预留了 // TODO 检索占位符。当团队准备好正式连入数据库时，只需：

在后端的 pom.xml 中引入 io.qdrant:client:1.8.0 依赖。

在预留代码处实例化 QdrantClient 连接本地的 6334 端口，传入用户的提问向量，即可获取 Top-K 的相关文档切片。

## 第二部分：前后端与 RAG 大模型交互链路

为了让团队成员在独立开发和联调时思路清晰，本模块的请求链路经过了严格的设计，确保前端解耦、跨域无障碍以及 API 密钥的安全。完整的交互流程如下：

### 1. 前端触发与跨域代理 (Vue 3 + Vite)

* **用户输入**：在独立的交互组件（如提供的参考示例 `RobotAgent.vue`）中，用户在输入框内输入关于物流或单证的问题并点击发送。
* **发起请求**：前端通过 `fetch` 或 `axios` 发起一个标准的 HTTP POST 请求，目标地址为 `/api/rag/chat`，请求体 (Body) 为 JSON 格式：`{ "query": "用户输入的问题" }`。
* **Vite 代理拦截**：为了避免浏览器在本地开发时报 CORS 跨域错误，`vite.config.js` 中配置了代理服务器。它会监听所有以 `/api` 开头的请求，并将其原封不动地转发到后端的 `http://localhost:8080`。

### 2. 后端接收与 RAG 检索 (Java Spring Boot)

* **接口接收**：请求到达 Spring Boot 代理层的 `LlmController` 中的 `POST /api/rag/chat` 接口，并自动被反序列化为 `RagRequest` 对象。
* **向量检索 (RAG)**：后端提取出用户的 `query`。在完整的 RAG 流程中，此处会先调用 Embedding 服务将文本转为向量，然后通过 HTTP/gRPC 查询本地的 Qdrant 数据库获取 Top-K 相关的上下文片段。
* **Prompt 组装**：后端将检索到的知识库片段拼接到 `system` 角色的提示词中，并把用户的原始问题放入 `user` 角色的提示词中，组装成符合规范的消息列表。

### 3. 云端大模型调用 (通义千问 DashScope)

* **读取配置**：`QwenProperties` 类从 `application.yml` (或本地覆盖的 `application-local.yml`) 中读取大模型的 `baseUrl` 和 `model`。
* **安全鉴权**：系统强制要求从环境变量读取真实的 `DASHSCOPE_API_KEY`（或从脱敏的本地配置文件中加载），并将其作为 `Bearer Token` 注入到 HTTP 请求的 `Authorization` Header 中，确保真实密钥绝不硬编码进前端或 Git 仓库。
* **发起调用**：后端使用 `RestTemplate` 将拼接好的对话历史发送到通义千问的 OpenAI 兼容接口 (`.../chat/completions`)。

### 4. 数据返回与渲染

* **后端解析**：Java 后端接收到阿里云返回的 JSON 数据后，提取出 `choices[0].message.content` 中的文本回答。
* **返回前端**：后端将提取出的文本重新封装为标准格式（如 `{ "success": true, "answer": "..." }`）返回给 Vite 代理，最终回到前端代码。
* **UI 更新**：`RobotAgent.vue` 接收到成功的响应后，将大模型的回答推入对话历史数组，并在聊天气泡 UI 中渲染展示给用户。

---
> **总结**：发给队友的核心是 `LlmController`、`QwenProperties`、`application.yml` 以及 `vite.config.js`。通过这套标准流程，你的本地向量检索代码就能完美无缝地嵌入到现有的 Spring Boot 架构中。
