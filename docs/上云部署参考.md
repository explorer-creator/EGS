# EGS 前端 + 代理上云部署参考

本文档面向 **EGSfb** 项目：`front`（Vue2 + Vite）与 `proxy`（Spring Boot 3 后端）。上云前请通读 **环境变量** 与 **xDM 相关约束**。

---

## 1. 架构说明

| 组件 | 本地开发 | 云上典型做法 |
|------|----------|----------------|
| 前端 | Vite 开发服务器（如 `5173`），`/api` 代理到后端 | 构建为**静态文件**，由 **Nginx / OSS+CDN / 平台静态托管** 提供 |
| 后端 | `proxy`，默认 **8080** | 容器或虚拟机中运行 **JAR**，平台分配端口或绑定 **80/443** |
| xDM-F | 本机 **8003**（可选） | 多数场景**无法直连**云上应用；见下文「xDM 与 Mock」 |

**请求路径**：浏览器 → 前端页面 → 同域或跨域请求 **`/api/**`** → Spring Boot →（可选）转发 xDM、调用地图/千问等。

## 2. 构建产物

### 2.1 前端

```bash
cd front
npm ci   # 或 npm install
npm run build
```

- 产物目录一般为 `front/dist`（以 `package.json` 的 `build` 脚本为准）。
- 生产环境 **不要** 依赖 Vite 的 `server.proxy`；需保证浏览器访问的 API 地址正确（见第 4 节）。

### 2.2 后端

```bash
cd proxy
mvn -DskipTests clean package
```

- 可执行 JAR 通常在 `proxy/target/*.jar`。
- 运行需 **JDK 17+**（与 Spring Boot 3.x 一致）。

```bash
java -jar target/xdm-f-gateway-service-1.0.0-SNAPSHOT.jar
```

（具体 JAR 名以 `target` 目录为准。）

## 3. 云上部署方式（任选）

### 方案 A：同一台机器 / 同一容器编排

- Nginx：`/api` 反代到 `127.0.0.1:8080`，`/` 指向 `dist`。
- 用户只访问一个域名，**无跨域问题**（或仅需简单配置）。

### 方案 B：前端与后端分离（不同子域或不同服务）

- 例：前端 `https://app.example.com`，后端 `https://api.example.com`。
- **必须**配置后端 **CORS**（见第 4 节），且前端 axios 的 `baseURL` 或请求 URL 需指向 API 域名（若当前为相对路径 `/api`，需改为完整 API 地址或通过网关统一前缀）。

### 方案 C：仅托管静态站 + 无服务器/容器 API

- 前端静态托管到 OSS、GitHub Pages、Vercel 等。
- 后端部署到云函数、容器服务、轻量应用服务器等，**开放 HTTPS**。
- 同样注意 **CORS** 与 **API 基地址**。

## 4. 环境变量与配置（重要）

以下与 `proxy/src/main/resources/application.yml` 对应，**上云时用环境变量覆盖**，勿把密钥写入仓库。

### 4.1 服务端口

| 变量 | 说明 |
|------|------|
| `SERVER_PORT` | 默认 `8080`，平台若指定端口则设置此项 |

### 4.2 xDM 代理

| 变量 | 说明 |
|------|------|
| `XDM_MOCK_MODE` | 建议云上 **`true`**（无 8003 时数据走内置 Mock，演示稳定） |
| `XDM_REMOTE_ENTITIES` | 逗号分隔实体名；非空时**仅**这些走真实 xDM，其余 Mock |
| `XDM_BASE_URL` | 若云上要连真实 xDM，填**可访问**的完整 services 根地址（含 `http(s)://` 与路径） |
| `XDM_APPLICATION_ID` / `XDM_TENANT_ID` / `XDM_USER_ID` / `XDM_AUTH_TOKEN` | 按你们 xDM 环境填写 |
| `XDM_REFERER` / `XDM_ORIGIN` | 与 xDM 控制台或网关允许的域名一致，常为 xDM 站点 Origin |

### 4.3 CORS（前后端不同源时）

| 变量 | 说明 |
|------|------|
| `CORS_ORIGINS` | 逗号分隔，例如 `https://你的前端域名` |

示例：

```bash
CORS_ORIGINS=https://app.example.com,https://www.example.com
```

### 4.4 其他业务

| 变量 | 说明 |
|------|------|
| `DASHSCOPE_API_KEY` | 千问（Qwen）API Key，**不要**写进 `application.yml` 提交仓库 |
| `AMAP_KEY` | 高德 Web 服务 Key（驾车路径等，服务端使用） |
| `CV_DEMO_MODE` | 视觉质检演示模式，云上演示可保持 `true` |
| `XDM_API_ENTITY_MAPPING` | 默认指向本机路径 `D:/EGS/...`，**Linux 云上需**改为实际挂载路径或挂载配置文件 |

## 5. Nginx 示例（同域反代）

```nginx
server {
    listen 443 ssl;
    server_name app.example.com;

    root /var/www/egs/dist;
    index index.html;

    location /api/ {
        proxy_pass http://127.0.0.1:8080/api/;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

- Vue 路由为 history 模式时，`try_files` 回退到 `index.html` 通常必要。
- 若前端构建时把 API 写死为绝对地址，需与 Nginx 的 `location` 一致。

## 6. xDM 与「上云」的现实情况

1. **云服务器无法访问你本机 8003**  
   - 默认 `XDM_BASE_URL` 指向 `127.0.0.1:8003` **在云上无效**（除非 xDM 也部署在同一 VPC 或专线可达）。

2. **推荐演示配置**  
   - `XDM_MOCK_MODE=true`，且 `XDM_REMOTE_ENTITIES` 留空（或只填云上能访问的实体）。  
   - 这样设备/物料/工序等 **Mock 正常**，无需 Cookie。

3. **必须连真实 xDM 时**  
   - xDM 需公网可达、或 VPN/专线、或把 xDM 部署在与 API **同一内网**；  
   - 同步修改 `XDM_BASE_URL`、`XDM_REFERER`、`XDM_ORIGIN`；  
   - 前端「Cookie」在浏览器侧配置，仍可能受 xDM 跨域与 Cookie 策略限制，需按 xDM 文档调试。

## 7. 安全清单

- [ ] 仓库中 **无明文** `DASHSCOPE_API_KEY`、数据库密码、生产 Cookie  
- [ ] 生产环境 **HTTPS** 与 **安全响应头**（由平台或 Nginx 配置）  
- [ ] 仅开放必要端口（如 80/443）；管理端口不暴露公网  
- [ ] 定期更新依赖与基础镜像（若使用 Docker）

## 8. Docker 思路（可选）

若需容器化，可自建多阶段镜像：

1. **Stage 1**：Node 构建 `front` → `dist`  
2. **Stage 2**：Maven 构建 `proxy` → JAR  
3. **运行镜像**：仅 JRE + JAR，或 JAR + Nginx 同镜像（也可用两个服务）

具体 `Dockerfile` 需按你选的云厂商（如镜像仓库、基础镜像源）微调，此处不绑定单一模板。

## 9. 常见问题

| 现象 | 排查方向 |
|------|----------|
| 前端白屏、路由 404 | Nginx `try_files`、前端 `publicPath` / `base` |
| 接口 `Network Error` / CORS | `CORS_ORIGINS`、浏览器是否 HTTPS 混用 HTTP API |
| 502 Bad Gateway | 后端未启动、端口、防火墙、健康检查路径 |
| xDM 数据为空 | 云上是否 `mock-mode`、是否误连不可达的 `8003` |
| `entity-mapping` 找不到 | Linux 上 `XDM_API_ENTITY_MAPPING` 路径与文件是否挂载 |

## 10. 部署后自检

1. 打开前端首页，能加载静态资源。  
2. 调用任意 **`/api`** 健康或业务接口（如设备列表），返回 200 且 JSON 正常。  
3. 在**仅 Mock** 配置下，设备/物料等页面可浏览、无依赖本机 8003。  
4. 若使用千问/高德，确认对应环境变量已在运行环境中生效。
