## 3. 当前「智能问答」实际怎么工作的（与向量库无关）

- **产品制造问答**、**小机器人**、**多模态报单**、**TMS 延误说明** 等：主要走 **Java 代理** → **DashScope 兼容接口（通义千问 / 视觉）**，属于 **直接 LLM 调用**，**检索增强（RAG）未实现**。
- **business-analysis** 子项目：LLM 走 **OpenAI 兼容 HTTP**，同样 **未见向量检索层**。

因此：**队友做向量库 = 新增能力**，不是「改现有库里某张表」。

---

## 4. 给队友的推荐技术路径（可写进方案书）

1. **选型（示例）**  
   - 轻量：**Chroma**、**Qdrant** 单容器；或与现有 **PostgreSQL** 用 **pgvector** 同栈。  
   - 规模：**Milvus**（需单独运维）。

2. **Embedding**  
   - 使用 **通义 / OpenAI 兼容** 的 **embedding 接口**（与现有 Key 体系对齐），或 **开源小模型** 本地部署。

3. **数据从哪来**  
   - 工序文档、物流 AI 接口说明、海关/贸易 **脱敏 FAQ**、产品手册片段等 → **切分 chunk** → **写入向量库**。

4. **和本仓库怎么连**  
   - **方案 A**：新建 `vector-service`（Python FastAPI 常见）：提供 `POST /query {question}` → 内部 **向量检索 + 拼 prompt + 调 LLM**。  
   - **方案 B**：在 **`proxy`（Spring Boot）** 增加 `/api/rag/chat`，由 Java 调向量服务再调千问。  
   - 前端：**RobotAgent**、**产品制造问答** 可增加「知识增强」开关，走新接口。

5. **不要和这些混淆**  
   - **`localStorage`**：仅前端演示缓存，**不适合** 存向量。  
   - **xDM / 8003**：若团队有主数据平台，可做 **元数据来源**，但向量索引仍需 **独立管道**。
